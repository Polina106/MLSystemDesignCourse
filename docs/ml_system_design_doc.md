# ML System Design Doc - Приложение для подбора рациона питания MVP 

## 1. Цели и предпосылки
Основная бизнес-цель проекта — решение проблемы составления разнообразного рациона питания клиентов при выбранной стратегии клиента (снижение веса, набор массы, поддержание веса). Уменьшение количества времени на планирование и составление рациона, поиска рецептов. 

### 1.1. Зачем идем в разработку продукта? Чем станет лучше при использовании ML?
1. **Решение многомерной задачи оптимизации с жесткими ограничениями**
Система может одновременно учитывать десятки ограничений и пожеланий (аллергии, калории, БЖУ, время приготовления, разнообразие), что практически невыполнимо для человека-диетолога при массовом обслуживании.

2. **Интеллектуальное обеспечение разнообразия блюд**
Статические планы питания или простые фильтры быстро приводят к однообразию, когда пользователь получает одни и те же блюда или ингредиенты несколько дней подряд. Система может целенаправленно максимизировать разнообразие, минимизируя повторение.

3. **Непрерывное улучшение и адаптация**
Система может отслеживать, какие рецепты пользователь отмечает как "понравившиеся", какие заменяет или пропускает. Эти данные используются для тонкой настройки модели ранжирования под каждого конкретного пользователя, делая будущие рекомендации все более точными.

При использовании машинного обучения сократится время поиска и подбора рецептов. План питания заранее будет учитывать аллергены, калории, время приготовления блюд и другие ограничения. Предложенный рацион будет персонализированным, построен будет на основе предпочтений пользователя. 

---

### 1.2. Бизнес-требования и ограничения

#### Основные бизнес-требования

* Персонализация питания - система должна учитывать индивидуальные параметры пользователя (возраст, вес, рост, активность) и цели (похудение/набор/поддержание)
* Автоматизация планирования - сокращение времени на составление рациона с 2-3 часов до 5-10 минут в день
* Соблюдение ограничений - 98% соответствие рациона заявленным аллергиям и непереносимостям
* Разнообразие рациона - минимизация повторения блюд до 15% в неделю для основных приемов пищи

#### Основные бизнес-ограничения

* Сроки пилота - 6 месяцев на разработку MVP
* Команда - 2 разработчика
* Инфраструктура - использование cloud-сервисов с месячным лимитом 10K руб.
* Работа в офлайне - модель работает в batch-режиме, прогноз отдается разово на день или на неделю 
* Качество данных - работа с существующей спарсенной базой рецептов без возможности массового рерайтинга

#### Ожидания от конкретной итерации

1. Итерация 1 (MVP - 3 месяца):
- Расчет калорийности по базовым параметрам (вес, рост, возраст, активность)
- Подбор рациона на день с учетом калорий и аллергий
- База из 200+ рецептов с корректным КБЖУ
- Web-интерфейс для заполнения анкеты и просмотра рациона
- Точность подбора калорий ±10% от целевого значения

2. Итерация 2 (2 месяца после MVP):
- Учет предпочтений "нравится/не нравится"
- План питания на неделю с автоматическим разнообразием

3. Итерация 3 (после валидации гипотез):
- ML-рекомендации на основе поведения пользователя
- Составление списка продуктов на неделю

### 1.3. Что входит в скоуп проекта/итерации, что не входит

#### На закрытие каких БТ подписываемся в данной итерации

1. Расчет суточной нормы калорий по базовым параметрам (вес, рост, возраст, активность)
2. Базовый подбор рациона с ограничениями
    - Учет аллергенов (исключение рецептов с запрещенными ингредиентами)
    - Соответствие целевой калорийности (±10%)
    - Баланс по приемам пищи: завтрак (±25%), обед (±35%), ужин (±25%), перекусы (±15%)
3. Content-based рекомендации, подбор разнообразных рекомендаций по признакам пользователя и рецептов
4. Constraint-based оптимизация дневного рациона, выбор оптимальных рецептов при учете ограничений по калорийности

#### Что не будет закрыто 

1. Сложные ML-алгоритмы рекомендаций
    - Нет коллаборативной фильтрации (требует данных о поведении пользователей)
    - Нет deep learning моделей (требует больших данных и вычислительных ресурсов)
    - Нет reinforcement learning для персонализации

2. Продвинутые фичи разнообразия
    - Автоматическое избегание повторения ингредиентов в течение недели
    - Учет сезонности продуктов
    - Баланс микронутриентов (бжу, витамины, минералы)

3. Интеграции и расширения
    - Нет мобильного приложения (только web-интерфейс)
    - Нет синхронизации с фитнес-трекерами

4. Оптимизация производительности
    - Реализация кэширования предвычисленных рекомендаций
    - Балансировка нагрузки для тысяч пользователей
    - Оптимизация запросов к базе рецептов

#### Описание результата с точки зрения качества кода и воспроизводимости решения

- Версионирование данных и кода
- Git репозиторий с понятной структурой коммитов
- DVC для версионирования датасетов рецептов
- `pyproject.toml` с зафиксированными версиями пакетов
- Использованы линтеры и форматеры
- Настроен pre-commit для проверки кода

### 1.4. Предпосылки решения

#### Данные и их источники 

1. База рецептов

Предпосылка: Используем существующую спарсенную базу из 500+ рецептов
Обоснование: 
- Быстрый старт без затрат на создание контента
- Достаточный объем для обеспечения разнообразия на MVP
- Возможность постепенного расширения и улучшения качества

Структура данных:
- Название рецепта, описание
- Ингредиенты с количествами
- Пищевая ценность (калории, белки, жиры, углеводы)
- Время приготовления, сложность
- Категории (тип блюда, кухня, теги)
- Аллергены и специальные пометки

2. Пользовательские данные

Предпосылка: Сбор через анкету при регистрации
Обоснование:
- Минимизация барьера входа (анкета на 3-5 минут)
- Достаточно для базовой персонализации
- Постепенное обогащение данных через feedback loop

Собираемые параметры:
- Демография: возраст, пол, вес, рост
- Цель: похудение/поддержание/набор массы
- Активность: уровень физической нагрузки
- Предпочтения: аллергии, нелюбимые продукты, время на готовку
- Ограничения: бюджет, доступные кухонные приборы

#### Временные горизонты

1. Горизонт прогнозирования: 1 день

Предпосылка: Генерация рациона на текущий день
Обоснование:
- Снижение сложности планирования
- Учет актуального состояния пользователя (возможны изменения планов)
- Постепенное обучение системы на ежедневном feedback

Развитие: 
- Итерация 2: планирование на 3 дня
- Итерация 3: недельное планирование с учетом закупок

2. Период переобучения моделей: 1 неделя

Предпосылка: Batch-обновление рекомендательных моделей
Обоснование:
- Баланс между адаптацией и стабильностью
- Техническая простота реализации на MVP
- Достаточный объем данных для переобучения за 1 неделю

Метрики для мониторинга:
- Изменение user engagement
- Качество предсказаний "лайков"
- Retention rate пользователей

#### Гранулярность модели

1. Уровень пользователя: индивидуальные рекомендации

Предпосылка: Персональные модели для каждого пользователя
Обоснование:
- Высокая вариативность пищевых предпочтений
- Индивидуальные метаболические особенности
- Разные цели и ограничения

Реализация:
- User-based features: демография, цели, предпочтения
- Персональные коэффициенты для расчета калорий
- Индивидуальные векторы предпочтений в content-based рекомендациях

2. Уровень рецепта: дискретные единицы планирования

Предпосылка: Работа с целыми рецептами, а не отдельными ингредиентами
Обоснование:
- Практичность для пользователя (готовые рецепты)
- Упрощение системы (меньше комбинаторики)
- Сохранение кулинарного контекста

Компромиссы:
- Меньшая гибкость в точной настройке КБЖУ
- Зависимость от качества рецептов в базе
- Возможность пропусков значений в пищевой ценности

## 2. Методология
### 2.1. Постановка задачи

**Основная техническая задача**: constraint-based оптимизация с элементами рекомендательной системы.

1. *Рекомендательная система*: Для каждого пользователя найти рецепты, максимально соответствующие его профилю предпочтений.

2. *Задача классификации и ранжирования*: Ранжирование рецептов по релевантности для конкретного пользователя и контекста.

3. *Задача Планирования*: Распределение рецептов по временным слотам дня с учетом:
- Временных ограничений (утро/день/вечер)
- Энергетических потребностей (калории по времени суток)
- Практичности приготовления

4. *Обработка многомерных ограничений*: Эффективная фильтрация и оптимизация в многомерном пространстве ограничений

### 2.3 Этапы решения

1. Подготовка данных
- Загрузка базы 500+ рецептов
- Очистка и нормализация данных
- Проведение EDA, обработка данных

2. Прогнозные модели
2.1 Cold Start (новые пользователи)
- Content-based фильтрация
- Демографические правила
- Популярные рецепты в группе

2.2 Пользователи с историей взаимодействий (когда наберется история взаимодействий)
- Content-based рекомендации
- История предпочтений
- Коллаборативная фильтрация

3. Constraint оптимизация
- Жесткие ограничения: аллергены, время готовки
- Целевые калории: ±10% от нормы
- Баланс БЖУ и приемов пищи
- Разнообразие рациона

4. Технический долг
- Кэширование Redis
- Genetic algorithm вместо greedy
- Мониторинг и логирование
- Обогащение базы рецептов

#### Этап 1. Сбор и подготовка данных

**Описание данных и сущностей**

На основе проведенного EDA выявлены 3 основных источника данных:

| Название данных | Есть ли данные в компании (если да, название источника/витрин) | Требуемый ресурс для получения данных (какие роли нужны) | Проверено ли качество данных (да, нет) |
| ------------- | ------------- | ------------- | ------------- |
| База рецептов, с рассчитанным каллоражем | Да, обучающий датасет | Data Scientist | + |
| Пары "юзер → рецепт" | Нет, датасет будет собран после MVP | Data Scientist / Data Engineer | + |
| База данных клиентов | Да, сгенерированный датасет | Data Scientist | + |

На первом этапе принято решение отказаться от использования в моделях источника данных по взаимодействиям между клиентом и рецептами, так как такие данные негде собрать. Модель будет работать сначала на рандомных рецептах, мы соберем фидбек клиентов. Далее обучим уже минимальную модель и итеративно будем дособирать данные и дообучать модели. Поэтому для такого источника данных будут необходимы компетенции Data Engineer, чтобы правильно организовать сбор и хранение данных.


**Объем данных**
База рецептов ~ 2500 записей. Стратификация для трейна и теста будет проходить по каллорийности рецептов, аллергенам (16 флагов) и времени приготовления. Для теста возьмем 20% выборки, время публикации рецепта не важно. 
База клиентов ~ 5000 записей. Стратификация будет сделана по полу, росту и весу.

**Репрезентативность данных**
После проведенного EDA было выявлено, что данные репрезентативны по калориям (есть рецепты для низкого, среднего и высокго уровня калорий), по аллергенам, по времени авктивного и общего приготовления. Даже проверено распределение данных по клиентам, представлены все весовые категори достаточно равномерно, есть представители со всеми видами аллергенов.

**Возможные риски**
Одним из ключевых рисков можно выделить - отсутствие аллергенов в списке, но наличие в рецепте. Данные парсились с сайта food.ru и аллергены проставлялись авторами вручную. То есть необходимо делать дополнительную проверку для выверки рецептов. 
Также в дальнейшем для обучения модели хорошего качество и выполнения условия разнообразия в определенных ограничениях может быть такое, что данных будет недостаточно. Можно расширить базу рецептов, спарсив данных с этого сайта (food.ru) или с аналогичных.

В данных нет конфиденциальной информации, которую необходимо было бы замаскировать, поэтому данный этап опустим.

**Feature engineering**
В качестве feature engineering я бы здесь использовала ручные методы. То есть добавление фичи - доля нутриентов в рационе, ане только абсолютное значение, преобразование времени приготовления в митуны. 
Для этапа с взаимодействиями я бы посчитала статистики для ингредиентов (доля рецептов с помидорами или рыбой, которая не понравилась). В целом как часто клиент ставит негативные и позитивные реакции. Также я бы добавила фичи по сегментам (например, бины по возрасту и полу).

**Необходимые результат этапа**
Необходимым результатом для данного этапа будем считать готовые данные по клиентам и по рецептам, которые можно использовать для первой версии моделирования.


#### Этап 2. Подготовка прогнозных моделей

**Общая структура**
Пайплайн моделирования состоит из 2 основных частей: Отбор кандидатов и решение задачи оптимизации.

1. Бейзлайн будет без данных про взаимодействия пользователей и рецептов, поэтому на этапе отбора кандидата будет учтена только фильтрация по аллергенам и по времени приготовления. 
MVP проекта на этой стадии содержит также модельную составляющую - косинусную схожесть между вектором рецептов и клиентов.

2. На втором этапе решается задача оптимизации по калориям. Есть 4 основных приема пищи и подбираются рецепты, которые в сумме будут составлять дневную норму калорий с погрешностью +- 5%. Для бейзлайна можно использовать простой greed search, проходится по топ-10 рецептах для каждого приема пищи из пераого этапа. Для более сложной версии можно использовать beem search или генетические алгоритмы.

Есть несколько версий подписок в приложении и от этого зависит необходимая частота обновления. То есть клиенты с вип-подпиской могут обновлять рацион хоть каждую минуту. Клиенты с бесплатной версией - один раз в сутки. 

**Функции потерь моделей**
Для первого этапа функций потерь нет, в MVP будет использоваться косинусное расстояние между вектором клиента и рецептами. Для улучшения точности также можно рассмотреть ANN подход, FAISS или HNSW. Далее при добавлении данных по взаимодействиям между клиентами и рецептами можно использовать модель ALS, например, с функцией потерь - MSE.

Для второго этапа функцией потерь будет MAE, так как важно именно абсолютное отклонение от заданной нормы. Далее можно улучшать и экспериментировать и использовать функции потерь более чувствительные к недопрогнозу или к перепрогнозу, в зависимости от выбранной стратегии (похудение, поддержание веса, набор веса).

**Оценка моделей**
1. Для первого этапа можно попробовать оценить расстояние между объектами по метрикам кластеризации (silhouette score).
2. Для оценки моделей второго этапа будут использованы стандартные метрики для рекомендательных систем - MRR, полнота, разнообразие, точность. Также будет учтена доля нутриентов (белков, жиров, углеводов), но именно как пост-оценка, а не составляющая функции потерь.

**Интерпретация моделей**
Интерпретировать модели достаточно сложно, однако для клиента можно выводить по шагам, сколько рецептов убираются по фильтрам сначала, например, и далее выводить процент уверенности модели.
Для разработки можно смотреть по каким именно признаках рецепт и клиент схожи на первом этапе.

**Возможные риски**
Ключевым риском в моделировании может стать недостаток рецептов для рекомендации в условиях выбранных ограничений. В таком случае можно добавить дополнительные рецепты, либо увеличить погрешность попадания в калорийность по 8%. Также можно увеличить кол-во рецептов, которые будут передаваться из первого этапа во второй до 15, например. 
Еще одним риском может быть долгая работа оптимизатора для подбора рецептов. Чтобы уменьшить количество времени можно выбирать иногда не самое оптимальное решение с помощью аппроксимирующих моделей, то есть использовать не жадный алгоритм, а тот же beem search.


#### Этап 3. Интеграция бизнес-правил 
Данный этап нужен для добавления специфических бизнес правил в работу модели. 
То есть необходимо определение разнообразия рациона, то есть сколько раз одно и то же блюдо может повторяться на неделе, например?
Пусть перекусы могут повторяться 2-3 раза, а полноценные блюда не более 2.

Также необходимо определить какой процент рецептов может не попадаться клиентам вообще?
Пусть этот процент будет равен 5%. Есть ряд очень калорийных блюд или блюд, которые требуют очень долгого приготовления. Возможно, такие блюда не будут актуальны в плане рациона.

Можно определить метрики точности для моделей при добавлении данных по взаимодействиям. 
Пусть общая точность (MAP) будет >80%.
